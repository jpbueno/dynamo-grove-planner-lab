Meet the Building Blocks

Grove introduces three nested abstractions that let you describe an entire inference pipeline in one place. Think of them as nesting dolls for GPU workloads.

PodClique: A Team of Identical Workers

🎭
Like players on a sports team

A basketball team has five starters, but if one gets injured, a substitute steps in. A PodClique works the same way — a group of identical pods filling the same role, with a minimum number that must always be available.

A PodClique is a group of pods with the same role. All decode workers form one PodClique. All prefill workers form another. The frontend is its own PodClique.

Each PodClique has two key numbers:

📈

replicas

Total desired pods. “I want 3 decode workers running.”

🛡

minAvailable

Minimum pods that must be running for the pipeline to function. “At least 1 decode worker must always be up.”

ScalingGroup: Teams That Move Together

Sometimes PodCliques need to scale as one unit. A prefill leader and its workers are useless without each other, so they belong in the same ScalingGroup.

🚂
Like carriages on a train

You can add more carriages to a train, but a locomotive and its dining car always move together. ScalingGroups bind dependent PodCliques so they expand and contract as one.

PodCliqueSet: The Whole Orchestra

The PodCliqueSet is the top-level object. It wraps all your ScalingGroups and PodCliques into one CRD that you apply with kubectl.

What This Looks Like in YAML

Here is a real DGDR manifest that deploys a model with SLA Planner support:

YAML 
apiVersion: nvidia.com/v1alpha1
kind: DynamoGraphDeploymentRequest
metadata:
  name: sla-planner-demo
spec:
  model: nvidia/Llama-3.1-Nemotron-Nano-8B-v1
  backend: vllm
  profilingConfig:
    config:
      sla:
        ttft: 1000.0
        itl: 100.0
PLAIN ENGLISH

Use NVIDIA's custom Kubernetes API (version v1alpha1).

This is a deployment request — a wish list for what you want running.

Give it a name so we can reference it later.

Call this deployment "sla-planner-demo".

Now describe what we actually want.

Serve this specific model from HuggingFace/NVIDIA.

Use vLLM as the inference engine behind the scenes.

Before deploying, profile the model to learn its performance characteristics.

SLA section — the latency promises we want to keep:

Each subsequent token must arrive within 100ms.
