The Orchestra Conductor

How Grove turns a YAML spec into running, topology-optimized pods — with every component in the right place, starting in the right order.

From Spec to Running Pods

🎼
Like a film director reading a screenplay

You hand the director a script (your PodCliqueSet YAML). The director doesn’t just hire actors — they decide where each scene is shot, which actors need to be on set together, and the exact order scenes are filmed. Grove is that director for your GPU cluster.

When you kubectl apply a Grove Operator picks up your PodCliqueSet and creates everything the pipeline needs:

1
Parse the Spec

Grove reads your PodCliqueSet and identifies all PodCliques, ScalingGroups, and their dependencies.

2
Create K8s Resources

Services, Secrets, HPAs, and pod templates are generated for each component.

3
Generate PodGangs

PodGangs define which pods must land together. The scheduler sees these as atomic units.

4
Schedule with Topology

The KAI Scheduler uses PodGangs to gang-schedule pods with GPU topology awareness.

Packed for Speed, Spread for Safety

Grove gives the scheduler two opposing topology goals — and the scheduler balances both:

📦

Network Packed

Within a replica: pods that talk to each other frequently are placed on the same node or NVLink domain. Minimizes latency between prefill and decode.

🔌

Network Spread

Across replicas: full copies of the pipeline are spread across different nodes or failure domains. If one node dies, the other replica keeps serving.

💡
Key Insight

This packed/spread duality is a universal infrastructure pattern. Databases do it (replicas on different availability zones). CDNs do it (edge servers spread globally). Grove brings it to GPU workloads.
