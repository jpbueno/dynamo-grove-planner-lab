The Autopilot

How Dynamo’s SLA Planner watches latency metrics in real time and scales prefill and decode workers independently to keep your promises.

Why “Set It and Forget It” Doesn’t Work

🌡
Like a thermostat, not a space heater

A space heater runs at full blast regardless of room temperature. A thermostat measures, predicts, and adjusts. The Planner is a thermostat for your inference latency — it doesn’t just deploy workers, it continuously tunes how many are running.

LLM traffic is bursty. A chatbot might see 5 requests per second at 3 AM and 500 at noon. Static GPU allocation means you either:

💰

Over-provision

Burn money on idle GPUs at 3 AM just so noon traffic doesn’t breach SLA.

🚨

Under-provision

Save money but breach latency targets when traffic spikes. Users see slow responses.

✅

Auto-scale

Let the Planner decide in real time. Alibaba ACK saw 80% fewer TTFT breaches with 5% fewer GPUs.

Offline Profiling + Online Scaling

The Planner works in two phases:

🔍
Phase 1: Offline Profiling

Before serving traffic, the Planner benchmarks the model on your hardware. It learns: “On an A100, this model can handle X requests per second at Y TTFT and Z ITL.” This creates an interpolation model — a lookup table of load-to-latency curves.

⚡
Phase 2: Online Scaling

Once live, the Planner collects real metrics from Prometheus every N seconds. It predicts future load using ARIMA or Prophet models, then checks: “At the predicted load, how many decode/prefill workers do I need to stay within SLA?”

Tuning the Planner for a Demo

The demo script overrides default Planner settings for faster feedback loops:

BASH 
OVERRIDES='{
  "adjustment-interval":"20",
  "itl":"10",
  "ttft":"50",
  "max-gpu-budget":"3",
  "load-predictor":"constant"
}'
PLAIN ENGLISH

Override the Planner defaults with demo-friendly values:

Check metrics every 20 seconds (default is 180s — too slow for a live demo).

Target ITL of 10ms per token (aggressively tight for demo visibility).

Target TTFT of 50ms (also tight — makes the Planner react quickly).

Allow up to 3 GPUs total (budget cap to prevent runaway scaling).

Use “constant” predictor instead of ARIMA (assumes current load continues, simpler for demo).

💡
Key Insight

In production, you would use ARIMA or Prophet for prediction and a 180-second interval. The tight demo values make the Planner react in seconds instead of minutes so you can see scaling happen live.
