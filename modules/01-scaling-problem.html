What If Your AI Could Scale Itself?

Why serving large language models broke the old Kubernetes playbook — and what NVIDIA built to fix it.

The Old Playbook: Copy-Paste Your Pod

For years, scaling a web service on Kubernetes was straightforward: you had one container, and if traffic grew, you just ran more copies of it.

📚
Think of it like a copy shop

Need more output? Just add more identical printers. Every printer does the same job, every page is the same. Simple, predictable, done.

This worked brilliantly for stateless web apps. But LLMs broke the model — because generating a single response actually involves two very different jobs.

Two Jobs, One Request

When you send a prompt to an LLM, the system does two fundamentally different things:

⚡

Prefill

Compute-bound. Reads your entire prompt in one pass and builds internal context. Needs raw GPU horsepower — like a sprinter doing a 100m dash.

🔄

Decode

Memory-bound. Generates tokens one at a time, reading from a KV Cache. Needs fast memory access — like a librarian pulling books off shelves.

💡
Key Insight

Because prefill and decode have opposite resource profiles, running them on the same GPU wastes capacity. Disaggregated inference puts them on separate GPUs so each can be tuned independently.

Why Vanilla Kubernetes Falls Short

Standard Kubernetes treats every pod as an independent unit. That assumption breaks with disaggregated inference:

1
Startup Order Matters

The frontend must be ready before workers register. Leaders before followers. K8s has no built-in way to enforce this across pods.

2
All-or-Nothing Scheduling

A decode worker without its prefill partner is useless. You need gang scheduling — all components land together, or none do.

3
Topology Awareness

Prefill and decode workers that communicate over NVLink need to be on the same node. Replicas for availability need to be on different nodes. K8s doesn't know about GPU network topology.

The Fix: Grove + Planner

NVIDIA built two components inside Dynamo to solve this:

🌱

Grove

A purpose-built Kubernetes API that deploys entire inference pipelines as a single unit — with gang scheduling, startup ordering, and topology-aware placement.

📊

SLA Planner

A real-time autoscaler that watches latency metrics and scales prefill and decode workers independently to meet your SLA targets.
