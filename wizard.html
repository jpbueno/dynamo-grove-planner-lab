⚡ Prefill COMPUTE-BOUND

Processes the entire input prompt in a single parallel pass, building the KV Cache. Needs raw GPU compute horsepower.

Analogy: A sprinter doing a 100m dash — intense burst of energy, then done.

🔄 Decode MEMORY-BOUND

Generates tokens one at a time by reading from the KV Cache. Needs fast, high-bandwidth memory access.

Analogy: A librarian pulling books from shelves one by one — steady, methodical work.

💡
Key Insight

Because prefill and decode have opposite resource profiles, disaggregated inference puts them on separate GPU pools — each independently scaled and optimized for its specific workload pattern.

Grove

Kubernetes API for Modern AI Inference Workloads

🏢
Airport Control Tower Analogy

Grove is like an airport control tower. It doesn't fly the planes (run inference), but it decides which gates they use, ensures they don't collide on the runway, and coordinates takeoffs and landings so the whole system runs smoothly. Without it, you'd have chaos — planes (pods) landing wherever there's space, with no regard for connections (data dependencies).

⚙ Features

📦

PodCliqueSet CRD

Custom resource that defines multi-component inference topologies as a single declarative unit.

👥

Gang Scheduling

All components of a replica land together, or none do. No half-deployed broken pipelines.

🌐

Topology-Aware Placement

Pack tightly-coupled workers on the same node (NVLink); spread replicas across nodes for HA.

📈

Multi-Level Autoscaling

Scale individual ScalingGroups or entire replicas based on independent metrics.

🔃

Startup Ordering

Frontend waits for workers; leaders before followers. Enforced sequencing across pods.

🔄

Smart Rolling Updates

Updates entire replicas at once instead of individual pods, preventing broken mid-update states.

🌐 Topology Rules

network: packed

Within a Replica

Components that exchange KV cache data (prefill → decode) are placed on the same node or within NVLink/NVSwitch domain for maximum bandwidth.

network: spread

Across Replicas

Independent replicas are spread across failure domains (different nodes, racks) to ensure high availability during node failures.

Planner

SLA-Driven Autoscaler for Disaggregated LLM Serving

🌡
Thermostat Analogy

The Planner is like a smart thermostat for your inference cluster. You set the desired "temperature" (SLA targets like TTFT < 50ms), and it continuously measures the actual conditions, predicts what's coming (weather forecast = load prediction), and adjusts heating/cooling (GPU replicas) to maintain comfort — never overshooting or wasting energy.

📅 Two Phases: Profile Once, Scale Forever

🔬 Offline Profiling

Profiles your model at various batch sizes on your specific GPU. Learns the throughput curve. AIConfigurator selects optimal config, then auto-deploys to Dynamo.

⚡ Online Scaling

Continuous control loop every --adjustment-interval seconds. Reads Prometheus, predicts load, PATCHes DGD replica count.

📊 Planner Architecture

🔗 AIConfigurator + Planner Workflow

Find the best initial setup offline with AIConfigurator, then automatically tune performance online with Planner.

🔬 Pre-Deployment Profiling

The Planner profiles your model across tensor-parallel (TP) configurations. Charts show throughput vs latency trade-offs — the Planner picks the configuration that meets your SLA targets at maximum throughput.

🔄 Planner Control Loop

1

Query Prometheus

Pull real-time metrics: request rate, P99 TTFT, P99 ITL, active batch sizes, queue depths.

2

Apply Correction Factors

Compare observed latency to SLA targets. If TTFT is 2x the target, correction factor > 1.0 (need more GPUs).

3

Predict Future Load

Use the configured load predictor (Constant, ARIMA, Kalman, or Prophet) to forecast traffic for the next interval.

4

Calculate Replicas

Map predicted load to GPU count using the offline profiling data. "This load needs N prefill + M decode GPUs."

5

Apply Constraints

Enforce --max-gpu-budget and --min-endpoint bounds. Never scale beyond what you've budgeted.

6

PATCH the DGD

Update the DynamoGPUDeployment resource with the new replica count. This triggers Grove reconciliation.

7

Reconcile

Grove picks up the DGD change, creates/destroys PodGangs, and the cluster converges to the desired state.

📋 Correction Factors

Factor ValueMeaningAction
≈ 1.0 Observed latency matches SLA target No change needed — system is well-tuned
> 1.0 Latency exceeds SLA target Scale up — add more GPU replicas
< 1.0 Latency well below SLA target Scale down — reclaim unused GPUs

📊 Load Predictors

PredictorBest ForComplexityNotes
constant Steady-state traffic Low Assumes current load continues. Good baseline.
arima Trending / seasonal traffic Medium Auto-regressive model. Needs historical data window.
kalman Noisy real-time signals Medium Filters out measurement noise. Fast convergence.
prophet Complex seasonal patterns High Facebook Prophet. Best for daily/weekly cycles.

💡
Separation of Concerns

Planner never touches pods directly — it only writes desired state to the DGD custom resource. Grove never reads Prometheus — it only reads the DGD and makes it real. This clean separation means you can swap out either component independently.

🏗 Full Architecture: Dynamo → Kubernetes → GPU Rack

The Planner (SLO-based) and Grove (Topology-optimized Serving) work together inside Dynamo. The Planner decides GPU allocation; Grove schedules pods as gangs onto the GB200 NVL72 rack with NVLink-aware placement.

❓ FAQ

DGDR (DynamoGPUDeploymentRequest) contains the model profiling data and SLA targets — it's the "wish list." DGD (DynamoGPUDeployment) is the actual deployment spec with concrete replica counts — it's the "execution plan." The Planner reads the DGDR and continuously writes/updates the DGD. Grove reads the DGD and creates the actual pods.
Any auto-regressive transformer model benefits from disaggregation. The larger the model and the more variable the traffic, the bigger the benefit. Small models with very low latency may not need it — the overhead of KV cache transfer could exceed the savings.
The model is run on a single GPU with synthetic traffic at increasing concurrency levels. For each level, the system records P50/P99 TTFT and ITL. This creates a throughput-latency curve that the Planner uses to map "if I have N req/s, I need M GPUs to stay within SLA."
Yes. Grove is a standalone Kubernetes operator. You can manually set replica counts in the DGD or use a different autoscaler. The Planner is an optional component that automates the "how many replicas" decision, but Grove handles all the scheduling, topology, and lifecycle management independently.
Grove uses an all-or-nothing approach: before creating any pods in a PodGang, it checks that all required resources (GPUs, memory, node affinity) are available. If any component can't be scheduled, the entire PodGang waits rather than partially deploying. This prevents the classic deadlock where Replica A's prefill holds GPUs waiting for decode, while Replica B's decode holds GPUs waiting for prefill.
Start with constant. It's the simplest and works surprisingly well for most workloads. If you have clear daily patterns, try arima. If your traffic is noisy (lots of spikes), kalman will smooth it out. Only use prophet if you have strong weekly/seasonal patterns and enough historical data.
Grove and the Planner are deployed in production at Alibaba Cloud's ACK (Container Service for Kubernetes) at significant scale. The results (80% fewer SLA breaches, 5% GPU reduction) come from real production workloads. NVIDIA continues to develop and harden both components as part of the Dynamo project.

📚 Glossary

Latency

TTFTTime To First Token. Measures how long before the model starts generating output after receiving a prompt.
ITLInter-Token Latency. The time between consecutive output tokens during the decode phase.
P9999th percentile. "99% of requests complete within this latency." The standard SLA measurement.
SLAService Level Agreement. The performance contract (e.g., "P99 TTFT < 50ms").

Architecture

PrefillThe compute-bound phase that processes the entire input prompt and builds the KV Cache.
DecodeThe memory-bound phase that generates tokens one-at-a-time using the KV Cache.
KV CacheKey-Value cache. Stores intermediate computations from prefill so decode doesn't recompute them.
Disaggregated InferenceArchitecture pattern that separates prefill and decode onto different GPU pools for independent scaling.

Planner

DGDRDynamoGPUDeploymentRequest. CRD containing model profiling data and SLA targets.
DGDDynamoGPUDeployment. CRD containing the concrete deployment spec with replica counts.
Correction FactorRatio of observed latency to SLA target. >1.0 means scale up, <1.0 means scale down.
Load PredictorAlgorithm that forecasts future traffic (constant, ARIMA, Kalman, Prophet).

Grove / Kubernetes

PodCliqueSetCRD that defines a multi-component inference topology as a single declarative unit.
PodCliqueA single role/pod in the inference pipeline (e.g., one prefill worker).
ScalingGroupA set of identical PodCliques that scale as a unit.
PodGangOne complete replica of all components, scheduled atomically (all-or-nothing).
Gang SchedulingEnsures all pods in a group are scheduled together or not at all.
Topology-Aware PlacementScheduling that respects GPU interconnect topology (NVLink, NVSwitch) and failure domains.
