The Problem

Why Vanilla Kubernetes Breaks for LLM Inference

Serving models isn’t just “N replicas of the same pod” anymore. Disaggregated inference demands coordination Kubernetes wasn’t built for.

🔥

The core challenge

  • Prefill is compute-bound; decode is memory-bound — they need different GPUs and different scaling
  • Standard K8s HPA scales on CPU/memory — it can’t reason about LLM latency (TTFT, ITL)
  • A decode worker without its prefill partner is useless — you need gang scheduling
  • Workers communicating over NVLink need topology-aware placement — K8s doesn’t understand GPU topology
  • Startup order matters: frontend before workers, leaders before followers — K8s has no built-in cross-pod ordering

⚡ Prefill

Compute-bound. Reads the entire prompt in one pass. Needs raw GPU horsepower — like a sprinter doing a 100m dash.

🔄 Decode

Memory-bound. Generates tokens one at a time from the KV Cache. Needs fast memory bandwidth — like a librarian pulling books off shelves.

💡 Key Insight

Running prefill and decode on the same GPU wastes capacity. Disaggregated inference puts them on separate GPUs so each is tuned for its bottleneck.

Product Deep Dives

The Components

Planner decides what to scale. Grove decides where and how to place it.

PLANNER
SLA-Driven Autoscaler for Disaggregated LLM ServingPart of NVIDIA Dynamo · Apache 2.0
Monitors real-time TTFT and ITL via Prometheus, uses profiling-backed capacity models, and independently scales prefill and decode workers to maintain your latency SLA.
💡Think of the Planner as a hospital triage system. It checks vital signs (TTFT/ITL metrics), predicts incoming patient load (request forecasting), and allocates the right specialists (prefill/decode workers) to meet care SLAs — all automatically.
  • 🎯 SLA Targets — Define TTFT and ITL in milliseconds. The Planner computes required replicas from profiling curves.
  • 📊 Online Profiling (AIPerf) — Automated benchmarking on your specific GPU. Outputs NPZ files consumed by the Planner.
  • 🔄 Correction Factors — Adapts when reality diverges from profiling. actual / expected compensates for prefix caching, prompt variance, hardware differences.
  • 📈 Four Load Predictors — ARIMA (trend), Kalman (smooth), Prophet (seasonal), Constant (mirror). Select via --load-predictor.
  • ⚙ Multi-Backend — vLLM, TensorRT-LLM, SGLang. Same Planner, different --backend flag.
  • 💰 GPU Budget Cap — --max-gpu-budget sets a hard ceiling the Planner never exceeds.
  • 👁 Observation Mode — --no-operation shows recommendations without scaling. Safe for validation.

The Planner Loop (every adjustment-interval seconds)

1. Query Prometheus → Get TTFT, ITL, request count, ISL, OSL
2. Compute correction factors → actual_latency / profiled_latency
3. Predict next interval load → ARIMA / Kalman / Prophet / Constant
4. Calculate required replicas → predicted_throughput / profiled_capacity
5. Apply constraints → max_gpu_budget, min_endpoint
6. PATCH DGD resource → Update replica counts
7. Operator reconciles → Scale Deployments up/down

GROVE
Kubernetes API for Modern AI Inference WorkloadsOpen Source · Apache 2.0 · github.com/ai-dynamo/grove
A single PodCliqueSet CRD defines your entire inference system — frontend, prefill, decode, planner — with roles, scaling groups, gang scheduling, topology placement, and startup ordering.
💡Think of Grove as an airport control tower. It coordinates gate assignments (topology placement), ensures all crew are ready before departure (gang scheduling), manages the departure sequence (startup ordering), and opens more runways at peak hours (multi-level autoscaling).
  • 🏗 PodCliqueSet CRD — Define frontend, prefill, decode, planner as roles in one resource. Grove creates Deployments, Services, HPAs, PodGangs automatically.
  • 🤝 Gang Scheduling — All pods in a scaling group are scheduled together or not at all. No partial deployments.
  • 🗺 Topology-Aware Placement — Generates constraints for KAI Scheduler to co-locate pods within NVLink domains. Replicas spread across nodes.
  • 📐 Multi-Level Autoscaling — Scale PodCliques, ScalingGroups, or PodCliqueSets independently.
  • 🔢 Startup Ordering — Declarative dependencies. Frontend → workers → planner.
  • 🔄 Smart Rolling Updates — Maintains component relationships during updates and recovery.

PodCliqueSet Hierarchy

PodCliqueSet (replicas=2)
├── Replica 0 — network packed for performance
│   ├── ScalingGroup "Frontend" (replicas=1, minAvailable=1)
│   │   └── PodClique: Frontend pods
│   ├── ScalingGroup "Prefill" (replicas=1, minAvailable=1)
│   │   └── PodClique: Prefill worker pods
│   └── ScalingGroup "Decode" (replicas=2, minAvailable=1)
│       └── PodClique: Decode worker pods
└── Replica 1 — network spread for availability
    └── (identical structure on different node)

FAQ

Frequently Asked Questions

Common questions from customers, partners, and internal teams.

Does the Planner work with aggregated (non-disaggregated) serving?+
Yes. The Planner can scale a single pool of workers based on SLA targets. Disaggregated mode unlocks independent prefill/decode scaling for better utilization, but it’s not required.
How long does profiling take?+
30–120 minutes depending on model size. One-time per model+GPU combo. DGDR automates it. For TRT-LLM/SGLang, AI Configurator can estimate in ~30 seconds via simulation database.
What if my traffic doesn’t match profiling assumptions?+
Correction factors handle this. Each cycle computes actual_latency / profiled_latency. If real prompts are shorter, the factor drops below 1.0 and fewer replicas are needed. Adapts continuously.
Can Grove work with a scheduler other than KAI?+
Yes. Grove generates PodGang resources consumed by KAI by default, but VirtualConnector mode lets you integrate with other schedulers.
What GPUs are supported?+
Any NVIDIA data center GPU: A100, H100, H200, GB200, L40S, and more. Profiling adapts automatically.
Can I use this for agentic pipelines?+
Yes. Grove’s PodCliqueSet can model any multi-component system — tool servers, RAG retrievers, routing layers — with scaling groups and startup ordering.
How does the Planner compare to KEDA or HPA?+
HPA/KEDA scale on CPU/memory/throughput. The Planner scales on latency SLAs with profiling-backed capacity models. It knows that to meet 50ms ITL at 200 req/s, you need exactly N decode replicas — KEDA can’t compute that.
Glossary

Key Terms

Quick reference for technical terms used throughout this course.

Latency & Metrics
TTFT
Time To First Token. How long from sending a prompt until the first response token appears.
ITL
Inter-Token Latency. Delay between consecutive response tokens. Low ITL = fluid streaming.
TPOT
Time Per Output Token. Total generation time divided by token count.
ISL / OSL
Input/Output Sequence Length. Average token counts used during profiling.
SLA
Service Level Agreement. Contractual performance thresholds (e.g., TTFT < 200ms).
Inference Architecture
Disaggregated Inference
Separates prefill and decode onto different GPU pools for independent scaling.
Prefill
First phase. Processes the entire prompt in one compute-heavy pass. Generates the KV Cache.
Decode
Second phase. Generates tokens one at a time from KV Cache. Memory-bandwidth bound.
KV Cache
Key-Value Cache. Stores intermediate computations so decode doesn’t recompute them.
NIXL
NVIDIA Inference eXchange Library. Handles efficient KV cache transfers between workers.
Planner Concepts
Correction Factor
Ratio of actual vs. profiled latency. Adapts to prefix caching, prompt variance, hardware differences.
ARIMA
AutoRegressive Integrated Moving Average. Statistical model for trend-based load prediction.
Interpolation Model
Lookup table mapping request rates to expected latency, built from profiling data.
DGDR
DynamoGraphDeploymentRequest. The YAML manifest to deploy a model with SLA targets.
DGD
DynamoGraphDeployment. The running deployment Grove manages after profiling completes.
Grove / Kubernetes
PodClique
Group of pods with same role/template (e.g., all decode workers). Scales independently.
ScalingGroup
Bundle of tightly-coupled PodCliques that scale as one unit.
PodCliqueSet
Top-level Grove CRD. Describes the full inference workload in a single resource.
PodGang
Scheduling constraint for atomic pod groups. All scheduled together or not at all.
Gang Scheduling
Ensures related pods all land or none do. Prevents half-deployed broken pipelines.
KAI Scheduler
NVIDIA’s AI-aware K8s scheduler with GPU topology and NVLink-aware placement.
NVLink
NVIDIA’s high-bandwidth GPU interconnect, 5–10x faster than PCIe.
GPU Time-Slicing
Multiple pods share one physical GPU by taking turns. Used in demos for scaling without extra hardware.
